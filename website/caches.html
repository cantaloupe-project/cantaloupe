Caches

Caches cache generated images on the server. They are pluggable, to enable different caching mechanisms.

Server-side caching is highly recommended, as it will drastically reduce load on the server and improve response times for the remaining processing that is left to do. There are other ways of caching on the server, but the Cantaloupe cache is simple and custom-tailored.

Enabling

Caching is optional, and disabled by default. To enable it, set cache.server to the name of a cache, such as FilesystemCache.

The cache can be configured to operate in one of two ways:

Conservative (cache.server.resolve_first = true)
Source images are looked up and verified to exist before cached representations are returned. This precludes returning a cached representation when the underlying resource no longer exists, but also impairs response times by a variable amount, depending on the resolver.
Aggressive (cache.server.resolve_first = false)
Cached representations are returned immediately, if available. This is faster, but it can lead to inconsistency between the cache and the underlying source image storage.

Notes

  • Caching can be very effective at improving performance, but it cannot reduce load to zero. Consider a caching reverse proxy and/or clustering when massive traffic is anticipated.
  • Currently, requests for full-sized, unaltered source images are not cached, and are instead streamed through with no processing.
  • Cantaloupe does not cache entire IIIF information response representations — only image dimensions, which are the only expensive part to generate. This means it is possible to change other configuration options that would affect the contents of information responses without having to purge the cache.
  • The cache is shared across API endpoints. Requests for the same image from different endpoints will return the same cached image.
  • Caches are thread-safe, but not process-safe. To avoid corruption, multiple simultaneously-running instances of Cantaloupe should not be pointed at the same cache.
  • Server-side caching can be augmented by client-side caching.

FilesystemCache

FilesystemCache caches generated images and (parts of) information requests into a filesystem directory. The location of this directory is configurable, as is the "time-to-live" of the cache files, with the following options:

FilesystemCache.pathname
Absolute path of the folder in which cached images and dimensions will be stored.
FilesystemCache.ttl_seconds
Time-to-live of the cached content, in seconds.

JdbcCache

JdbcCache caches generated images and (parts of) information requests into tables in a relational database (RDBMS). This cache can be configured with the following options:

JdbcCache.connection_string
JDBC connection string; for example, jdbc:postgresql://localhost:5432/mydatabase.
JdbcCache.user
User to connect to the database as.
JdbcCache.password
Password to use when connecting to the database. Can be left blank if not needed.
JdbcCache.image_table
The table in which to cache images. This will be created automatically.
JdbcCache.info_table
The table in which to cache information responses. This will be created automatically.
JdbcCache.ttl_seconds
Time-to-live of the cached images and information responses, in seconds.
Note: JdbcCache has been tested with PostgreSQL 9.4 and H2 1.4. Other databases may work, but are untested. In particular, the database needs to support CREATE TABLE IF NOT EXISTS syntax, which in the case of PostgreSQL, arrived in version 9.1.
Note: JdbcCache is inefficient in that it buffers generated images in memory before flushing them to the database, as opposed to streaming them. (This is currently unavoidable, as the JDBC API couples poorly with the Cantaloupe cache API.)

Purging the Cache

To purge expired images only, start Cantaloupe with the -Dcantaloupe.cache.purge_expired option.

To purge all images, start Cantaloupe with the -Dcantaloupe.cache.purge option.
