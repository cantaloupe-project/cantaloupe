Source Formats

Various Cantaloupe processors support several other formats than those listed here; what follows are mainly just notes.

JPEG

It should be possible to use TurboJPEG (high-level API over libjpeg-turbo) with Java2dProcessor to hardware-accelerate JPEG coding. To do so, include the TurboJPEG plugin JAR in your classpath, and compile libjpeg-turbo with Java support (using the --with-java configuration option). (The author has not tried this.)

JPEG2000

JPEG2000 uses advanced compression techniques to enable fast reduced-scale and region-of-interest decoding. In theory, and with a performant decoder, JPEG2000 is well-suited for extremely large source images.

KakaduProcessor is by far the most efficient processor for this format, and it performs very well, even with images in the range of hundreds of megapixels.

OpenJpegProcessor uses the OpenJPEG decoder, which is generally considered, as of this writing, to be the fastest open-source JPEG2000 decoder. (ImageMagickProcessor's JPEG2000 delegate, if installed, will also use the OpenJPEG library, but less efficiently as it won't use OpenJPEG's level reduction feature.)

GraphicsMagickProcessor can read and write JPEG2000 using JasPer, if the necessary plugin is installed. This will probably not be fast enough to be usable for most purposes.

See JaiProcessor for notes on the mediaLib accelerator.

TIFF

Most processors can read this format, though some have trouble with its special cases. Additionally, there are some conditions that TIFF images must meet in order to be delivered efficiently—read on for details.

Strip-Based vs. Tile-Based

The Adobe TIFF 6.0 specification permits arrangements of image data in either strips or tiles. Most TIFF encoders produce strip-based TIFFs unless told to do otherwise. These work fine at small sizes, but are increasingly slow to read as size increases. High-resolution TIFFs must be tile-based in order to permit efficient region extraction. An easy way to check is with the tiffdump utility:

$ tiffdump image.tif

For strip-based TIFFs, this will print out some information including StripOffsets, StripByteCounts, and so on. For tile-based TIFFs, it will print TileOffsets, TileByteCounts, and so on, instead.

Multi-Resolution (Pyramidal) TIFF

Multi-resolution TIFF is a special type of TIFF file that can improve reader efficiency at reduced scales. In addition to the main image, a multi-resolution TIFF file will contain a sequence of progressively half-sized sub-images: for example, a 10000×10000 pixel main image would include derivatives of 5000×5000 pixels, 2500×2500 pixels, 1250×1250 pixels, and so on, all embedded inside the same file.

Each of the levels in a multi-resolution TIFF file is, itself, a TIFF image that can be striped or tiled just like a mono-resolution TIFF file. Tiled and pyramidal encoding are complementary: the former improves efficiency with reduced regions at large scales, and the latter improves efficiency at reduced scales.

Summary

For efficient deep zooming, TIFF images need to be pyramidal, and each level of the pyramid should be tiled. The reduced-scale pyramid levels will be used at small scales (shallow zoom levels), while the tiles of the large-scale levels will be used at large scales.

Java2dProcessor and JaiProcessor both support multi-resolution TIFF, which is to say that they actually do read the embedded sub-images and choose the smallest one that can fulfill the request. Java2dProcessor additionally exploits tiled sub-images, for the best of both worlds.
